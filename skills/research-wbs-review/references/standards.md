# WBS编制标准与审核准则

## WBS编制的四大核心原则

### 1. 分类组合原则

WBS由多个WBE（Work Breakdown Element）组成，每个WBE必须属于组织认可的标准类型。

**WBE标准类型**:
- **目标**: 一级WBE，代表可交付成果（硬件、软件、报告、标准、数据集等）
- **里程碑**: 重要的时间节点或阶段性成果
- **正常**: 常规工作任务
- **低风险/中风险/高风险**: 根据风险等级分类

**审核要点**:
- 所有一级WBE必须标记为"目标"类型
- WBE类型必须在标准类型范围内
- 如需新类型，必须经组织批准

### 2. 厘清目标原则

WBS的形成过程是"科研任务、预期目标、输入输出关系、责任边界不断厘清"的过程。

**审核要点**:
- 每个WBE必须有清晰的任务描述
- 一级WBE应明确标注交付物类型（如：交付物-系统/产品、交付物-报告、交付物-数据集等）
- 关键任务必须明确开始和结束日期
- WBE的描述应能清晰反映：
  - 要做什么（任务内容）
  - 交付什么（输出成果）
  - 如何衡量（完成标准）

**命名规范建议**:
- 一级WBE: `交付物-[类型]/[具体名称]`
  - 例如：`交付物-系统/产品A`、`交付物-调研报告/市场分析`
- 二级及以下：清晰描述模块/组件/功能名称

### 3. 双方确认原则

WBS反映科研项目的实施方案、研究内容、技术路线、关键技术等要素，应由双方共同协商确认。

**审核要点**:
- 所有一级WBE必须明确责任单位/团队/人员
- WBS结构和分解逻辑应双方认可
- 责任边界应清晰无争议
- 交付标准和验收方式应明确

### 4. 责任匹配原则

WBE分解的"颗粒度"应与追责问效的责任单元相匹配。

**审核要点**:
- 同一个可交付的WBE只能由一个责任单位/团队/人员承担
- 叶子节点（最底层WBE）应明确责任人
- 颗粒度应适中：
  - 不能太粗：无法有效追责
  - 不能太细：增加管理成本
- 一般下一级WBE由上一级WBE负责人确定
- 责任分配规则应透明

**责任分配检查**:
- Level 1 (一级WBE): 必须有责任单位/团队
- Level 2-3 (子任务): 建议明确责任人
- Leaf nodes (叶子节点): 必须有明确责任人

## WBS结构完整性要求

### 层级编号规范

**编号格式**:
- Level 1: `1`, `2`, `3` ...
- Level 2: `1.1`, `1.2`, `2.1` ...
- Level 3: `1.1.1`, `1.1.2`, `2.1.1` ...
- Level 4: `1.1.1.1`, `1.1.1.2` ...

**审核要点**:
- WBS编号的层级数应与Level字段一致
- 编号应连续，无跳跃
- 同级编号应唯一

### 必填字段检查

| 字段 | Level 1 | Level 2+ | Leaf Node |
|------|---------|----------|-----------|
| WBS编号 | 必填 | 必填 | 必填 |
| 任务描述 | 必填 | 必填 | 必填 |
| Type | 必填(目标) | 可选 | 可选 |
| 责任人 | 必填 | 建议 | 必填 |
| 开始日期 | 建议 | 建议 | 建议 |
| 结束日期 | 建议 | 建议 | 建议 |
| Status | 可选 | 可选 | 建议 |

### 状态一致性

**标准状态值**:
- 未分配
- 未开始
- 进行中
- 延误
- 完成

**审核要点**:
- 状态值必须在标准范围内
- 已完成的任务应有结束日期
- 进行中的任务应有开始日期

## 常见问题与改进建议

### 问题1: 一级WBE描述不规范
**表现**: 描述为"其它工作"、"杂项"等模糊名称
**影响**: 无法识别交付物类型，不利于管理和评价
**建议**: 使用"交付物-[类型]/[名称]"格式，明确说明交付成果

### 问题2: 责任分配不清
**表现**: 叶子节点无责任人，或多个WBE共享责任人但职责不明
**影响**: 出现问题时无法追责
**建议**: 确保每个可执行的WBE都有唯一明确的责任人

### 问题3: 时间规划缺失
**表现**: 缺少开始/结束日期
**影响**: 无法进行进度跟踪和风险预警
**建议**: 为所有关键WBE设定时间节点，尤其是一级和叶子节点

### 问题4: 颗粒度不当
**表现**: 过粗（一级WBE直接分解到具体任务）或过细（4-5层以上）
**影响**: 管理成本过高或追责粒度不足
**建议**: 一般控制在3-4层，根据项目复杂度调整

### 问题5: WBS结构不完整
**表现**: WBS编号跳跃、层级关系混乱
**影响**: 结构不清晰，难以理解项目全貌
**建议**: 检查编号连续性和层级关系，确保结构完整

## 审核流程建议

1. **结构检查**: 验证WBS编号、层级关系的正确性
2. **分类检查**: 验证WBE类型是否符合标准
3. **目标检查**: 验证任务描述、交付物、时间节点的完整性
4. **责任检查**: 验证责任分配的合理性和完整性
5. **一致性检查**: 验证状态、日期等字段的逻辑一致性
6. **双方确认**: 确认WBS已经双方协商并达成共识

## 审核报告格式

审核报告应包含:
1. **基本信息**: 文件名、WBE总数
2. **问题统计**: 按严重程度分类（高/中/低）
3. **详细问题列表**: 按原则分类，包含：
   - 问题位置（行号、WBS编号）
   - 问题描述
   - 改进建议
4. **总体评价**: 是否通过审核，主要改进方向
