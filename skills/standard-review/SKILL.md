# 标准文件审校 (Standard Document Review)

基于 GB/T 1.1-2020《标准化工作导则 第1部分：标准化文件的结构和起草规则》，对标准化文件进行全面的符合性审查和质量检查。

## 审校工作流程

### 第一步：了解审校需求

与用户确认以下信息：
1. **文件类型**：国家标准、行业标准、地方标准、团体标准还是企业标准？
2. **文件格式**：.doc/.docx、.md、.pdf 还是其他格式？
3. **审校深度**：
   - **快速审查**：仅检查结构完整性和必备要素
   - **标准审查**：检查结构、要素、条款类型和能愿动词（推荐）
   - **深度审查**：全面检查包括编排格式、图表规范等
4. **重点关注**：是否有特别需要关注的方面？

### 第二步：读取文件内容

根据文件格式使用适当的工具读取：
- **.docx 文件**：使用 Read 工具读取
- **.md 文件**：使用 Read 工具读取
- **.pdf 文件**：使用 Read 工具读取（可能需要 OCR）

**分段读取策略**（针对大文件）：
1. 先读取封面、目次、前言部分
2. 读取范围、规范性引用文件、术语和定义
3. 读取主要技术章节
4. 读取附录、参考文献部分

### 第三步：执行审校检查

根据选定的审校深度，按以下检查清单进行：

#### 3.1 结构完整性检查（所有审校深度）

**必备要素检查**：
- [ ] 封面
- [ ] 前言
- [ ] 范围
- [ ] 规范性引用文件（章节号和标题）
- [ ] 术语和定义（章节号和标题）
- [ ] 核心技术要素

**可选要素检查**：
- [ ] 目次
- [ ] 引言
- [ ] 符号和缩略语
- [ ] 分类和编码/系统构成
- [ ] 总体原则和/或总体要求
- [ ] 其他技术要素
- [ ] 附录
- [ ] 参考文献
- [ ] 索引

**要素顺序检查**：
标准顺序应为：封面 → 目次 → 前言 → 引言 → 范围 → 规范性引用文件 → 术语和定义 → 符号和缩略语 → 其他技术要素 → 附录 → 参考文献 → 索引

#### 3.2 层次结构检查（标准审查及以上）

**层次划分规范性**：
- [ ] 章编号从1开始连续（范围、规范性引用文件、术语和定义...）
- [ ] 条编号使用下脚点（5.1、5.1.1、5.1.1.1）
- [ ] 条的层次不超过5层
- [ ] 列项符号正确：
  - 第一层：破折号（——）或 a)、b)、c)
  - 第二层：间隔号（·）或 1)、2)、3)

**常见问题**：
- ❌ 使用"5-1"代替"5.1"
- ❌ 条的层次超过5层
- ❌ 章编号不连续
- ❌ 列项符号混乱

#### 3.3 文件名称检查（标准审查及以上）

**名称构成检查**：
- [ ] 主体元素是否明确（必备）
- [ ] 引导元素是否必要（可选）
- [ ] 补充元素是否合理（可选）
- [ ] 是否包含功能类型词语（术语、符号、分类、编码、试验方法、规范、规程、指南）
- [ ] 是否避免了"···标准""···国家标准"等字样

**参考**：见 [elements-structure.md](references/elements-structure.md) 第6.1节

#### 3.4 条款类型和能愿动词检查（标准审查及以上）

这是标准审校的**核心内容**。详细规则见 [clause-types.md](references/clause-types.md)

**快速检查方法**：
1. 使用 Grep 工具搜索能愿动词：
   ```
   pattern: "(应|不应|宜|不宜|可|不必|能|不能|可能|不可能|必须|不得|不可|禁止)"
   ```

2. 逐一检查每个能愿动词的使用是否正确

**重点检查项**：

| 能愿动词 | 正确用法 | 常见错误 |
|---------|---------|---------|
| 应/不应 | 表示强制要求 | ❌ 使用"必须"代替"应" |
| 宜/不宜 | 表示推荐 | ❌ 混淆要求和推荐 |
| 可/不必 | 表示允许 | ❌ 使用"能""可能"代替"可" |
| 能/不能 | 表示能力 | ❌ 使用"可"代替"能" |
| 可能/不可能 | 表示可能性 | ❌ 使用"可"代替"可能" |

**严禁使用的词语**：
- ❌ "必须"（应改为"应"）
- ❌ "不得"、"不可"、"禁止"（应改为"不应"）

**定性要求检查**：
- ❌ "应足够坚固"（应具体化）
- ❌ "应较为便捷"（应具体化）

#### 3.5 主要要素内容检查（标准审查及以上）

**范围检查**：
- [ ] 使用陈述句，不使用要求型或推荐型条款
- [ ] 简明陈述文件涵盖的主题
- [ ] 必要时说明适用和不适用范围

**规范性引用文件检查**：
- [ ] 引导语是否规范
- [ ] 文件是否按编号顺序排列
- [ ] 注日期引用是否正确标注年份
- [ ] 无引用文件时是否说明"本文件没有规范性引用文件"

**术语和定义检查**：
- [ ] 术语条目结构完整（编号+术语+定义，可选：英文对应词、示例、注）
- [ ] 定义准确、简明、自洽
- [ ] 避免循环定义
- [ ] 使用陈述句

**前言检查**：
- [ ] 包含起草规则声明
- [ ] 包含版本关系说明（如适用）
- [ ] 包含归口单位信息
- [ ] 包含起草单位和起草人
- [ ] **不应包含要求型条款**
- [ ] **不应包含推荐型条款**

**附录检查**：
- [ ] 正确标明"(规范性)"或"(资料性)"
- [ ] 位于正文之后、参考文献之前
- [ ] 编号使用大写拉丁字母（附录A、附录B）
- [ ] 附录内图、表、公式编号正确（A.1、表A.1、图A.1）

#### 3.6 引用和提示检查（深度审查）

**规范性引用检查**：
- [ ] 要求型条款引用使用"应符合"
- [ ] 试验方法引用使用"按""按照"
- [ ] 文件编号完整（代号+顺序号+年份号）

**资料性引用检查**：
- [ ] 使用"见""给出"等词语
- [ ] 资料性引用的文件列入参考文献

**提示自身内容检查**：
- [ ] 规范性提示使用适当能愿动词+内容编号
- [ ] 资料性提示使用"见"+内容编号
- [ ] 避免"见上文""见下文"等模糊表述

#### 3.7 图表和公式检查（深度审查）

**图的检查**：
- [ ] 图编号从1开始连续
- [ ] 图编号和图题置于图下方
- [ ] 标引序号使用正确
- [ ] 单位标注规范

**表的检查**：
- [ ] 表编号从1开始连续
- [ ] 表编号和表题置于表上方
- [ ] 每个表有表头
- [ ] 单位标注规范（表头中或右上角"单位为..."）
- [ ] 表头不使用斜线

**数学公式检查**：
- [ ] 公式编号右端对齐
- [ ] "式中:"后正确解释符号
- [ ] 符号一致性（同一符号表示同一量）

#### 3.8 其他规范性检查（深度审查）

**注和示例**：
- [ ] 单个注用"注:"，多个用"注1:"、"注2:"
- [ ] 单个示例用"示例:"，多个用"示例1:"、"示例2:"
- [ ] 注和示例位置正确

**脚注**：
- [ ] 条文脚注全文连续编号（1)、2)、3)...）
- [ ] 图表脚注每个单独编号（a、b、c）
- [ ] 条文脚注在页面左下方

**常用词使用**：
- [ ] "和"与"或"使用正确
- [ ] "包括"与"由...组成"使用正确
- [ ] 其他常用词无误用

### 第四步：生成审校报告

使用报告模板 [review-report-template.md](assets/review-report-template.md) 生成详细的审校报告。

**报告结构**：
1. 审校总体评价
2. 结构完整性审查
3. 层次结构审查
4. 文件名称审查
5. 条款类型和能愿动词审查
6. 主要要素内容审查
7. 引用和提示审查
8. 图表和公式审查
9. 其他规范性审查
10. 编排格式审查
11. 问题汇总
12. 审校结论

**报告生成要点**：
- 问题分级：严重问题（必须修改）、一般问题（建议修改）、建议改进
- 每个问题注明具体位置（章节号）
- 提供具体的修改建议
- 使用符号：✅（符合）、❌（不符合）、⚠️（基本符合）

### 第五步：与用户沟通

1. **呈现报告**：提供完整的审校报告
2. **重点说明**：突出严重问题和主要建议
3. **答疑解惑**：解答用户关于审校结果的疑问
4. **协助修改**（如需要）：帮助用户修改发现的问题

## 审校技巧

### 高效审校方法

1. **使用 Grep 工具快速定位**：
   - 搜索能愿动词：`(应|不应|宜|不宜|可|不必|能|不能|可能|不可能|必须|不得|不可|禁止)`
   - 搜索章节号：`^(#{1,4})\s+(\d+\.?)+`
   - 搜索图表编号：`(图|表)\s*\d+`

2. **分模块审校**：
   - 先整体后局部
   - 先结构后内容
   - 先必备后可选

3. **重点关注**：
   - 能愿动词使用（最常见的问题）
   - 必备要素完整性
   - 层次编号规范性

### 常见问题快查

| 问题类型 | 快速检查方法 |
|---------|------------|
| "必须"误用 | Grep 搜索 `必须` |
| "不得/不可/禁止"误用 | Grep 搜索 `(不得\|不可\|禁止)` |
| 章编号不规范 | 检查是否有 `1-1`、`1.1.1.1.1.1` 等 |
| 列项符号混乱 | 检查 `——`、`a)`、`·`、`1)` 使用 |
| 范围使用要求型 | 检查范围章节是否有"应" |
| 前言包含要求 | 检查前言是否有"应""宜" |

## 注意事项

1. **审校深度与时间**：
   - 快速审查：适合初稿或时间紧张
   - 标准审查：推荐用于正式审查
   - 深度审查：用于报批前终审

2. **专业术语**：
   - 尊重专业领域的术语习惯
   - 不过度纠正专业术语的表述

3. **客观中立**：
   - 基于 GB/T 1.1-2020 客观评价
   - 不主观臆断作者意图
   - 问题明确、建议具体

4. **报告质量**：
   - 问题描述准确、具体
   - 修改建议可操作
   - 重点突出、层次清晰

## 快速开始示例

**用户**："请审校这个标准文件"

**Claude**：
```
我将按照 GB/T 1.1-2020 的要求为您审校这个标准文件。

请先确认以下信息：
1. 文件类型：国家标准/行业标准/地方标准/团体标准/企业标准？
2. 审校深度：快速审查/标准审查（推荐）/深度审查？
3. 特别关注：有没有特别需要关注的方面？

同时请提供文件路径或内容。
```

然后根据用户需求，执行审校流程，最后生成详细的审校报告。
